## 🟡 MEDIUM — Multiple Permissive RLS Policies (Performance + Security)

Multiple `OR`-joined policies on the same table/role/action degrade query performance and make access logic harder to reason about.

**Critical table — `jobs` (all 4 DML operations affected):**
- DELETE: `Tradies manage jobs for own contacts` + `Users can delete own jobs`
- INSERT: `Clients can create jobs` + `Tradies manage jobs for own contacts`
- SELECT: `Tradies manage jobs for own contacts` + `Users can view relevant jobs`
- UPDATE: `Tradies manage jobs for own contacts` + `Users can update relevant jobs`

**Other affected tables:**
- `invoices` (SELECT + UPDATE): duplicate billing user policies
- `time_entries` (SELECT): 6 role variants all hitting 2 policies
- `business_team_members` (INSERT): 2 insert policies
- `recurring_jobs` (SELECT): 2 select policies
- `recurring_sessions` (SELECT): 2 select policies
- `site_visit_events` (SELECT): client + tradie select policies

**Recommendation:** Consolidate into single policies using `OR` within the policy body rather than separate policies. Create new migrations — do not edit existing ones.

Ref: https://supabase.com/docs/guides/database/database-linter?lint=0006_multiple_permissive_policies

## 🟡 MEDIUM — Unindexed Foreign Key

`public.client_contacts.client_contacts_linked_profile_id_fkey` has no covering index. Queries joining `client_contacts` to `profiles` via `linked_profile_id` will do full table scans.

**Fix:**
```sql
-- In a new migration:
CREATE INDEX idx_client_contacts_linked_profile_id
  ON public.client_contacts(linked_profile_id);
```

Ref: https://supabase.com/docs/guides/database/database-linter?lint=0001_unindexed_foreign_keys
