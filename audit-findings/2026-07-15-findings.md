## 🆕 NEW — `imported_calendar_visits` RLS Auth Init Plan (Performance)

The `owner manages imported visits` policy on `public.imported_calendar_visits` (added in migration `20260713000000_google_calendar_import.sql`) re-evaluates `auth.uid()` for **every row** rather than once per query. This causes full table rescans at scale.

**Fix** — create a new migration `supabase/migrations/20260715001000_fix_calendar_visits_rls_initplan.sql`:

```sql
-- Replace re-evaluating auth.uid() with a subquery (evaluated once per query)
DROP POLICY IF EXISTS "owner manages imported visits" ON public.imported_calendar_visits;

CREATE POLICY "owner manages imported visits"
  ON public.imported_calendar_visits
  FOR ALL
  TO authenticated
  USING ((select auth.uid()) = owner_id)
  WITH CHECK ((select auth.uid()) = owner_id);
```

Ref: https://supabase.com/docs/guides/database/database-linter?lint=0003_auth_rls_initplan

## 🟡 MEDIUM — Multiple Permissive RLS Policies (12 instances)

Multiple OR-joined policies on the same table/role/action degrade query performance.

**Highest impact — `jobs` table (all 4 DML ops):**
- DELETE: `Tradies manage jobs for own contacts` + `Users can delete own jobs`
- INSERT: `Clients can create jobs` + `Tradies manage jobs for own contacts`
- SELECT: `Tradies manage jobs for own contacts` + `Users can view relevant jobs`
- UPDATE: `Tradies manage jobs for own contacts` + `Users can update relevant jobs`

**Other affected tables:**
- `invoices` SELECT + UPDATE: duplicate billing-user policies (added by migration `20260708140000`)
- `time_entries` SELECT: 6 role variants (anon, authenticated, authenticator, cli_login_postgres, dashboard_user, supabase_privileged_role) — 2 policies each
- `business_team_members` INSERT: 2 policies
- `recurring_jobs` SELECT: 2 policies (added by `20260705120000`)
- `recurring_sessions` SELECT: 2 policies
- `site_visit_events` SELECT: 2 policies (added by `20260706102000`)
