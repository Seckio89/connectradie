## 🟢 RESOLVED SINCE YESTERDAY

### Google Calendar OAuth State Parameter — HMAC-Signed (Fixed July 12)
The `google-calendar-oauth` edge function was refactored to use a cryptographically signed state parameter (`HMAC-SHA256`). Previously the state was a bare user UUID which could be forged — any attacker knowing a victim's UUID could bind their own Google account to the victim's calendar integration. The new implementation:
- Signs `userId.expiresAtMs` with `OAUTH_STATE_SECRET` (falls back to service role key)
- Verifies the signature using `crypto.subtle.verify` (constant-time)
- Enforces a 10-minute TTL on the OAuth flow
- Narrows the Google OAuth scope to `calendar.events` only (also committed July 12)

This is a meaningful security improvement.

### Payouts.tsx innerHTML — Partially Mitigated (July 12)
The `safeTemplateSrc` custom template URL is now validated against an allowlist (`data:image/...` or `https://`) and HTML-escaped before interpolation. The main `html` variable is React-rendered DOM (escaped at render time). Risk is now **low** for the template injection vector. The `container.innerHTML = finalHtml` assignment remains but is now considered safe given the sanitization in place. No further action required unless raw DB strings are interpolated into `html` outside React (verify on next major edit to Payouts.tsx).

## 🆕 NEW — `time_entries` Multiple Permissive SELECT Policies

**Root cause:** Migration `20260711140000_worker_timesheet_visibility.sql` added a new `"Workers can view own time entries"` policy for SELECT on `public.time_entries`, overlapping the existing `"Business owners can manage time entries"` policy.

Supabase now flags this for 6 different DB roles (`anon`, `authenticated`, `authenticator`, `cli_login_postgres`, `dashboard_user`, `supabase_privileged_role`). Both policies run on every SELECT query to `time_entries`, which will add query overhead as the table grows.

**Fix:** Merge the two SELECT policies into one using `OR` logic:
```sql
-- In a new migration:
DROP POLICY IF EXISTS "Business owners can manage time entries" ON public.time_entries;
DROP POLICY IF EXISTS "Workers can view own time entries" ON public.time_entries;

CREATE POLICY "time_entries_select" ON public.time_entries
  FOR SELECT USING (
    -- Business owner path
    EXISTS (
      SELECT 1 FROM public.business_team_members btm
      WHERE btm.business_id = time_entries.business_id
        AND btm.user_id = auth.uid()
        AND btm.role = 'owner'
    )
    OR
    -- Worker own entries path
    time_entries.user_id = auth.uid()
  );
```
