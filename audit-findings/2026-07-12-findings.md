## 🔴 HIGH — Supabase: Anon-Executable SECURITY DEFINER Functions

These 5 functions run with elevated DB privileges AND can be called by **unauthenticated** users via the REST API. Unless intentional (e.g. public business search), these should revoke anon EXECUTE or switch to SECURITY INVOKER.

| Function | Endpoint | Notes |
|---|---|---|
| `public.get_daily_profile_view_count(viewer_uuid uuid)` | `/rest/v1/rpc/get_daily_profile_view_count` | Anon can query profile view counts — likely unintended |
| `public.get_service_worker_details(p_recurring_job_id uuid)` | `/rest/v1/rpc/get_service_worker_details` | Exposes worker details to anon if UUID is known |
| `public.get_team_site_activity(p_since timestamp)` | `/rest/v1/rpc/get_team_site_activity` | Team activity readable without auth |
| `public.has_user_engagement(user_uuid uuid)` | `/rest/v1/rpc/has_user_engagement` | Engagement metadata readable without auth |
| `public.search_businesses_by_name(search_term text)` | `/rest/v1/rpc/search_businesses_by_name` | May be intentional for public search — verify |

**Fix:** For each unintended one, create a migration:
```sql
REVOKE EXECUTE ON FUNCTION public.get_daily_profile_view_count(uuid) FROM anon;
REVOKE EXECUTE ON FUNCTION public.get_service_worker_details(uuid) FROM anon;
REVOKE EXECUTE ON FUNCTION public.get_team_site_activity(timestamptz) FROM anon;
REVOKE EXECUTE ON FUNCTION public.has_user_engagement(uuid) FROM anon;
```

## 🟡 MEDIUM — innerHTML XSS Risk (PDF Generation)

Two files construct off-screen DOM elements with `.innerHTML` for PDF export using `html2pdf.js`. The content injected includes user-controlled database values (business name, bill-to name, addresses, notes) that are not HTML-escaped before being embedded in string templates.

**If a malicious user stores an XSS payload in an invoice field, it could execute in the victim's browser when they download a PDF.**

### `src/components/InvoiceViewModal.tsx:172`
```ts
const printContents = printRef.current.innerHTML;
container.innerHTML = `<div style="..."> ${printContents} </div>`
```
The `printContents` value is already React-rendered HTML (which does escape values), so risk here is lower — React's own XSS protection applies at render time. **Low-medium risk.**

### `src/pages/Payouts.tsx:595`
```ts
container.innerHTML = finalHtml;
```
Verify `finalHtml` is constructed from sanitized/pre-rendered React output only. If any raw database string is template-interpolated into `finalHtml`, this is a direct XSS vector.

**Recommended fix:** Audit all template interpolations in `Payouts.tsx` PDF generation path. Consider using `DOMPurify.sanitize()` on any user-derived HTML strings before assigning to `innerHTML`.

## 🟡 MEDIUM — Multiple Permissive RLS Policies (Performance)

These tables have multiple permissive policies for the same role+action, causing every policy to be evaluated per query. This is a correctness risk if policies were meant to be mutually exclusive:

| Table | Role | Action | Policies |
|---|---|---|---|
| `time_entries` | `anon`, `authenticated`, others | SELECT | "Business owners can manage time entries" + "Workers can view own time entries" |
| `jobs` | `authenticated` | SELECT / INSERT / UPDATE / DELETE | "Tradies manage jobs for own contacts" + base policies |
| `invoices` | `authenticated` | SELECT / UPDATE | "Billed users..." + "Users can view accessible invoices" |
| `business_team_members` | `authenticated` | INSERT | "Business owner can insert" + "Members can request to join" |
| `recurring_jobs` | `authenticated` | SELECT | Two overlapping policies |
| `recurring_sessions` | `authenticated` | SELECT | Two overlapping policies |
| `site_visit_events` | `authenticated` | SELECT | Client + tradie policies |

**Recommended action:** Consolidate overlapping policies using `OR` conditions within a single policy per role+action where possible.

Reference: https://supabase.com/docs/guides/database/database-linter?lint=0006_multiple_permissive_policies
